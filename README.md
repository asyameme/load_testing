# load_testing


## Этап 1: Выбор средств, установка и начальная конфигурация

**Хостовая машина:**

- ОС: **macOS 26.3.1**
- ЦП: **Apple M4**
- ОЗУ: **16 ГБ**
- доступное дисковое пространство: **245 ГБ**

**Гипервизор:**

- Oracle VirtualBox Manager 7.2.6

  Данный гипервизор был выбран по нескольким причинам:
  - не требует сложной начальной настройки
  - предоставляет удобный графический интерфейс для создания и управления виртуальными машинами
  - позволяет явно задавать параметры виртуального оборудования
  - поддерживает режим NAT и механизм port forwarding

**Гостевые ОС:**

- Ubuntu 24.04.4 LTS ARM64
- Windows 11 Home 25H2 ARM64

Эксперимент проводился в среде Apple Silicon, поэтому обе гостевые ОС были выбраны в ARM-совместимых редакциях

**Параметры обеих ВМ:**

- 2 vCPU
- 4096 МБ RAM
- 50 ГБ virtual disk (dynamic allocation)
- NAT

**СУБД:**

- PostgreSQL 16.13 на обеих ВМ

Причины выбора СУБД:
- поддерживается HammerDB
- широко используемая реляционная СУБД
- можно установить как на Linux, так и на Windows, сохранив одну и ту же версию продукта на обеих виртуальных машинах

На обеих ВМ
- для обеспечения сетевого доступа в `postgresql.conf` задан параметр  
    `listen_addresses = '*'`
- в `pg_hba.conf` добавлено правило  
    `host all all 0.0.0.0/0 scram-sha-256` для разрешения удаленных TCP/IP-подключений к PostgreSQL с парольной аутентификацией


**Инструмент нагрузочного тестирования:**
- **HammerDB CLI v5.0**

**Размещение HammerDB:**
- контейнер на хостовой машине
- контейнерный runtime: **Podman**

**Проброс портов:**
- Ubuntu VM: **5433 → 5432**
- Windows VM: **5434 → 5432**


## Этап 2: Сравнительное тестирование производительности под разными ОС

Цель этапа: определить, как тип гостевой операционной системы влияет на производительность базы данных при прочих равных условиях.

Параметры нагрузочного теста:
- Тип теста: TPROC-C
- Количество виртуальных пользователей: 2
- Время разогрева: 2 минуты
- Длительность основного теста: 10 минут


В качестве основных метрик производительности использовались показатели TPM (Transactions Per Minute) и NOPM (New Orders Per Minute), выдаваемые HammerDB

Для повышения надёжности результатов каждый сценарий был выполнен три раза.

В рамках проведенных тестов Ubuntu продемонстрировала заметно более высокую производительность, чем Windows. 
Linux-гостевая система обеспечила примерно в 1.95 раза больше транзакций в минуту. Аналогичная картина наблюдается и по метрике NOPM  --  преимущество Ubuntu составило примерно 1.93 раза.

При почти двукратном превосходстве по производительности Ubuntu потребляла меньше процессорного времени. Это может указывать на меньшие накладные расходы ОС и более эффективную обработку нагрузки в рассматриваемой конфигурации. Также Ubuntu в данном эксперименте показала более экономичное использование памяти.

По итогам сравнения для следующего этапа работы была выбрана Ubuntu 24.04.4 LTS ARM64, так как эта ОС показала лучшие результаты в базовой конфигурации виртуальной машины.

## Этап 3: Тестирование с изменением параметров ВМ

В рамках серии экспериментов сравнивались три конфигурации:

- 1 vCPU / 4 ГБ RAM
- 2 vCPU / 4 ГБ RAM
- 4 vCPU / 4 ГБ RAM

| Конфигурация ВМ   |       TPM |      NOPM | CPU ср., % | RAM ср., MB | CPU iowait ср., % | Disk util ср., % | Disk write ср., kB/s | Наблюдения                                                             |
| ----------------- | --------: | --------: | ---------: | ----------: | ----------------: | ---------------: | -------------------: | ------------------------------------------------------------------------- |
| **1 vCPU / 4 ГБ** | **70844** | **30753** |   **34.6** |     **927** |          **35.4** |         **39.4** |            **19932** | Производительность заметно ниже, чем при 2 vCPU; одного ядра недостаточно |
| **2 vCPU / 4 ГБ** | **91739** | **39566** |   **30.9** |     **946** |          **32.8** |         **59.6** |           **67072** | Наилучший баланс между производительностью и выделенными ресурсами        |
| **4 vCPU / 4 ГБ** | **91647** | **39725** |   **15.1** |     **993** |          **17.3** |         **50.7** |            **42031** | Прирост относительно 2 vCPU практически отсутствует                       |


Полученные результаты показывают, что увеличение числа виртуальных ядер с 1 до 2 приводит к заметному росту производительности: значение TPM увеличилось с 70844 до 91739, а NOPM увеличилось с 30753 до 39566. Дальнейшее увеличение числа виртуальных ядер с 2 до 4 уже не привело к заметному росту производительности, что указывает на достижение точки насыщения, после которой добавление процессорных ресурсов уже не даёт существенного эффекта.
Во всех конфигурациях заметную роль продолжала играть дисковая подсистема. Для 1 vCPU средний iowait составлял около 35.4%, для 2 vCPU — около 32.8%, для 4 vCPU — около 17.3%. Даже при снижении iowait в конфигурации с 4 vCPU это не привело к существенному росту итоговых значений HammerDB
